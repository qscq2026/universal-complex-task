# Universal Complex Task Automation Framework

> A structured, human-in-the-loop framework for managing **complex multi-step tasks** within a single Claude Code session.

[![Version](https://img.shields.io/badge/version-v3.1-blue)]()
[![Architecture](https://img.shields.io/badge/architecture-Factory--inspired-green)]()

## Overview

This framework implements a **multi-role simulation** of a factory production line — Orchestrator, Worker, Validator — all within a **single Claude Code instance**. It transforms vague or underspecified requirements into systematically executed, verified deliverables.

Designed for tasks that meet **2+ of these criteria**:

| Dimension | What it means |
|-----------|---------------|
| 📁 File scope | ≥3 files or cross-module changes |
| 🧠 Knowledge domains | Multi-domain intersection (code + architecture + test + docs) |
| 🔀 Path certainty | Multiple possible implementations, needs exploration |
| 🔗 Hidden dependencies | Changes to A cascade to B/C/D |
| 🔄 Completion rounds | Requires multi-turn persistence |
| ✅ Verification dimensions | Multi-dimensional validation (functional + performance + security) |

### Core Formula

```
T = (U ⊙ S ⊙ C) × O × [ (W → Cm → H → V → F)^k ]^m × A × M
```

| Symbol | Meaning |
|--------|---------|
| **T** | Task Output — final deliverable |
| **U** | User Intent — natural language input |
| **S** | Structured Spec — human-readable specification |
| **C** | Validation Contract — machine-executable assertions |
| **⊙** | Freeze Operator — once confirmed, cannot be reshaped |
| **O** | Orchestrator — plans, decomposes, schedules, never implements |
| **W** | Worker — single-feature implementer |
| **Cm** | Commit / Snapshot — immutable rollback point per feature |
| **H** | Handoff — structured handover document |
| **V** | Validator — independent inspection, fresh context |
| **F** | Fix Feature — generated by O when validation fails |
| **k** | Convergence iterations within a milestone |
| **m** | Milestone count — stage-gated, independently deliverable |
| **A** | Audit Log — full-chain traceability |
| **M** | Memory — cross-task experience accumulation |

## Architecture

```
                  ┌─────────────────────────────┐
                  │        Orchestrator (O)      │
                  │  Plan · Decompose · Schedule  │
                  └──────────┬──────────────────┘
                             │
           ┌─────────────────┼─────────────────┐
           │                                   │
    ┌──────▼──────┐                    ┌───────▼───────┐
    │  Worker (W) │                    │ Validator (V) │
    │  Implement  │                    │  Inspect      │
    └──────┬──────┘                    └───────┬───────┘
           │                                   │
           └─────────────┬─────────────────────┘
                         │
                  ┌──────▼──────┐
                  │  Handoff(H) │
                  │  Structured │
                  │  Markdown   │
                  └─────────────┘
```

All three roles are **context switches within the same session** — no separate processes. Communication between roles is exclusively via structured files (Handoffs, shared state, Audit Log). **Direct peer-to-peer chat between roles is prohibited.**

## Pipeline Phases

### Phase 0 — Pre-Check
Evaluate the task against the 6 complexity dimensions. If < 2 flags, switch to fast-execution mode.
If ≥ 4 of the 6 flags are triggered, the task is flagged as **unusually complex** — this unlocks an optional Coverage Clarification Pass (Phase 1.2b) to help surface gaps in the spec before the human confirms it.

### Phase 0.5 — State Detection & Resume
Before any initialization, check for existing task state. If a previous session was interrupted, present resume options to the user:
- **Resume** — continue from exact last state
- **Restart** — discard and start fresh
- **Inspect** — review detailed state before deciding

### Phase 1 — Goal Definition (U → S → C)
Three layers of specification, each with human confirmation:

1. **U Parsing** — Extract explicit + implicit requirements, write to `01-user-intent.md`
2. **S Generation** — Structured spec with functional/non-functional/anti-requirements + verification requirements. Features **guess-first ambiguity handling**: defaults are filled with industry-standard choices (hard cap: 3 `[NEEDS CLARIFICATION]` markers), and every assumption is recorded for structured confirmation → **Human confirms**
   - **Coverage Clarification Pass** (Phase 1.2b, optional, only for tasks flagged unusually complex at Phase 0): A structured 5-category gap assessment (Scope & Boundaries · Success Definition · Structural Dependencies · External Constraints · Verification Approach) with targeted questions, coverage table, and deferred/outstanding resolution flow.
3. **C Generation** — Validation contract with machine-executable assertions → **Human spot-checks**
4. **Feature Planning** — Milestone breakdown with assertion mapping → **Human confirms**

### Phase 2 — Workspace Initialization
Create task workspace with `init-task.sh`, initializing `state.json` and all reference templates.

### Phase 3 — Milestone Execution Loop
For each milestone (m iterations):
- **M-i.1** O: Milestone kickoff — load state, handoff, memory; generate feature plan
- **M-i.2** Feature Execution Loop (k iterations per feature):
  - **W** — Worker implements one feature
  - **Cm** — Commit snapshot
  - **H** — Structured Handoff written
  - **V** — Validator inspects with fresh context
  - **F** — O generates fix if validation fails; loop until pass
- **M-i.3** Milestone Gate — memory sweep, state update, human checkpoint

### Phase 4 — Audit Log
Cross-cutting append-only stream recording every action (who/when/what/why/result).

### Phase 5 — Memory & Final Delivery
Project-level memory accumulation, final validation sweep, deliverable assembly.

### Phase 6 — Graceful Termination
Prevent runaway loops — global iteration budget, convergence check, human termination signal.

### Phase 7 — Suspend & Resume
User-initiated suspend preserves exact execution context for future resume.

## Quick Start

### Via Claude Code Directly

```
/universal-complex-task "refactor the authentication module to support OAuth2,
split into 3 milestones, with TDD and integration tests"
```

The skill will auto-populate all files; you only need to confirm at **3 human checkpoints** during Phase 1.

### Using init-task.sh

```bash
# Clone this repo into your Claude Code skills directory
# Then initialize a task workspace:
./scripts/init-task.sh my-task-name

# With explicit project root:
PROJECT_ROOT=/path/to/project ./init-task.sh my-task-name

# Auto-confirm the detected root:
CONFIRM_ROOT=1 ./init-task.sh my-task-name
```

> The script creates an empty workspace for manual requirement drafting. For full automation, call the skill directly via Claude Code.

## Design Principles

| Principle | Description |
|-----------|-------------|
| **🧊 Freeze Operator** | Once U→S→C is confirmed by human, spec cannot be reshaped mid-implementation |
| **🔄 Serial Writes** | Only one writer at a time — no merge conflicts |
| **👁️ Fresh Context** | Validator reads contract + committed artifact only; no implementation bias |
| **📋 Structured Handoff** | No conversation memory passed between roles — only structured markdown |
| **📎 Append-only Audit** | Full traceability — every action recorded, never deleted |
| **🧠 Memory Pre-Load** | Cross-task lessons loaded before roles act, not just logged after |
| **🛑 No Self-Correction** | W never fixes its own bugs — O generates fix tasks from V findings |

## Relationship to Factory

This framework is **inspired by** the Factory multi-agent architecture but adapted for a **single-instance** environment:

| Aspect | Factory | This Framework |
|--------|---------|---------------|
| Processes | Multi-model/process | Single Claude Code instance |
| Roles | Separate agents | Context switches within one session |
| Communication | Agent-to-agent chat | Structured files only |
| Concurrency | Parallel | Serial writes, parallel reads |
