# Universal Complex Task Automation Framework v3.2

## Formula
T = (U ⊙ S ⊙ C) × O × [ (W → Cm → H → V → F)^k ]^m × A × M

## Symbol Definitions
- T: Task Output — deliverable final result
- U: User Intent — natural language input
- S: Structured Spec — human-readable specification (human confirms Layer 1)
- C: Validation Contract — machine-executable assertions (human spot-checks, O/V contract)
- ⊙: Freeze Operator — once U→S→C is confirmed by human, cannot be reshaped by implementation
- O: Orchestrator — plans/decomposes/schedules/accepts; never touches implementation details
- W: Worker — single-feature implementer; serial writes; starts with Fresh Context
- Cm: Commit / Snapshot — version snapshot after each feature; immutable rollback point
- H: Handoff — structured markdown (completed/remaining/commands/issues); replaces conversation memory
- V: Validator — independent inspection; Fresh Context; reads only Contract + committed artifact
- F: Fix Feature — generated by O when validation fails; never self-corrected
- k: Convergence — iterations within a single Milestone (Factory: 2-4 validation rounds)
- m: Milestone — stage-gated waterfall nodes; each independently deliverable
- A: Audit Log — full-chain audit (who/when/what/why/result); cross-cutting stream
- M: Memory — cross-task/cross-round experience accumulation (reusable templates + anti-patterns)

---

## Architecture Declaration
This skill simulates Factory's multi-agent pattern within a single Claude Code instance:
- O, W, V roles are context switches within the same session, not separate processes
- All inter-role communication is via structured files (H, shared state, Audit Log)
- Direct Communication (peer-to-peer chat between roles) is PROHIBITED
- Writes are serial; reads may be parallel

## Phase 0: Pre-Check — Is This Complex?

Evaluate against 6 dimensions. If < 2 "complex" flags, switch to fast-execution mode.

| Dimension | Simple | Complex |
|-----------|--------|---------|
| File scope | <= 2 files | >= 3 files or cross-module |
| Knowledge domains | single | multi-domain intersection |
| Path certainty | unique path | multiple paths, needs exploration |
| Hidden dependencies | none | cascade impact likely |
| Completion rounds | 1-2 turns | requires multi-turn persistence |
| Verification dimensions | single | multi-dimensional |

## Phase 0.5: State Detection & Resume (Lightweight Lifecycle)

Before any initialization, O MUST check for existing task state.

### 0.5.1 Task Directory Existence Check
- If `.agent/tasks/{task-id}/` exists:
  - Read `state.json`
  - If `status` is INIT / CONFIRMED / RUNNING / SUSPENDED → **Enter Resume Flow**
  - If `status` is COMPLETED → Ask user: "Task already completed. Start new task or append as change request?"
  - If `status` is DELETED → Treat as new task
- If directory does not exist → Proceed to normal initialization

### 0.5.2 Resume Flow (O Self-Regulated)
1. O reads `state.json` to reconstruct exact execution context:
   - `phase`: which Milestone/Feature/Iteration
   - `fix_depth`: current fix chain depth
   - `global_iteration`: consumed global budget
   - `completed`: what's already done
   - `frozen`: which files are locked
   - `last_role` + `last_action` + `last_result`: what happened right before interruption
2. O reads the last Handoff, Verification, and Audit Log entries to verify consistency
3. O writes a `RESUME` entry to Audit Log with full state snapshot
4. O presents to user:
   ```
   Detected unfinished task: {task-id}
   Status: {status} | Phase: {phase} | Feature: {current_feature}
   Last action: {last_role} {last_action} → {last_result}
   Completed: {completed_milestones} / {total_milestones}

   [Resume] Continue from where it left off
   [Restart] Discard progress and restart from Milestone 1
   [Inspect] Show detailed state before deciding
   ```
5. If user chooses Resume → O continues from exact state in `state.json`
6. If user chooses Restart → O archives current task to `.agent/tasks/archive/`, then starts fresh
7. If user chooses Inspect → O dumps full `state.json` + last 10 Audit Log entries + last Handoff

### 0.5.3 State Persistence Rules
- `state.json` is updated by O **after every single action** (not just at milestones)
- Frozen files (`01-04`) are NEVER overwritten during resume
- Audit Log is append-only; resume adds `RESUME` row, never deletes history
- Memory (`.agent/memory.md`) is read during resume but not modified until task completes

## Phase 1: Goal Definition (U → S → C) with Human-in-the-Loop

### 1.1 U Parsing
1. Extract explicit requirements from user input
2. Infer implicit requirements (backward compatibility, performance, security, team conventions)
3. Identify anti-requirements (what NOT to do) to prevent over-engineering
4. Write to `.agent/tasks/{task-id}/01-user-intent.md`
5. **Update state.json**: `status: INIT`, `phase: U_PARSED`

### 1.2 S Generation (Structured Spec) — Human Confirm Layer 1
1. O produces a human-readable structured specification including:
   - Scope summary (one sentence)
   - Functional requirements (numbered)
   - Non-functional requirements (performance, security, compatibility)
   - Anti-requirements (explicit exclusions)
   - Assumptions & dependencies
2. Write to `.agent/tasks/{task-id}/02-structured-spec.md`
3. **Update state.json**: `status: INIT`, `phase: S_GENERATED`, `frozen: []`
4. **HUMAN CHECKPOINT 1**: Present S to user; wait for confirmation or revision
   - Phrase: "This is my understanding of your goal. Please confirm or correct."
5. Once S is confirmed, it is FROZEN (⊙ operator)
6. **Update state.json**: `status: CONFIRMED`, `phase: S_CONFIRMED`, `frozen: ["02"]`

### 1.3 C Generation (Validation Contract) — Human Spot-Check Layer 2
1. O translates S into machine-executable assertions BEFORE any implementation
2. Each assertion format:
   ```
   ### ASSERT-{ID}: {Behavior Description}
   - Trigger: {precondition}
   - Action: {user/system action}
   - Expected: {observable outcome}
   - Tool: {verification tool}
   - Evidence: {required evidence type}
   - Maps to Spec: {S requirement number}
   ```
3. Write to `.agent/tasks/{task-id}/03-validation-contract.md`
4. **Update state.json**: `phase: C_GENERATED`
5. **HUMAN CHECKPOINT 2**: Show assertion count + coverage matrix (assertion → S requirement)
   - Phrase: "I derived N assertions from the spec. Here is the coverage matrix. Please spot-check or approve."
6. Once C is confirmed, it is FROZEN and cannot be reshaped by implementation
7. **Update state.json**: `phase: C_CONFIRMED`, `frozen: ["02", "03"]`

### 1.4 Feature Planning & Assertion Mapping — Human Confirm Layer 3
1. O decomposes S into Features (F-1, F-2, ...)
2. Each Feature must include:
   - ID, name, input, output, acceptance criteria, dependencies, risk level, estimated effort (in rounds)
3. O produces an Assertion Coverage Map: every assertion must be covered by >= 1 Feature
4. Group Features into Milestones (M-1, M-2, ...) forming a waterfall
   - Each Milestone is independently deliverable and verifiable
5. Write to `.agent/tasks/{task-id}/04-features-and-milestones.md`
6. **Update state.json**: `phase: PLAN_GENERATED`
7. **HUMAN CHECKPOINT 3**: Confirm feature list + milestone plan
   - Phrase: "These features cover all assertions. Milestones are staged as shown. Approve to proceed?"
8. Once approved, plan is FROZEN
9. **Update state.json**: `status: CONFIRMED`, `phase: PLAN_CONFIRMED`, `frozen: ["02", "03", "04"]`

## Phase 2: Workspace Initialization

Create `.agent/tasks/{task-id}/` with:
- `01-user-intent.md` (U)
- `02-structured-spec.md` (S) — FROZEN after confirmation
- `03-validation-contract.md` (C) — FROZEN after confirmation
- `04-features-and-milestones.md` (plan) — FROZEN after confirmation
- `05-audit-log.md` (A) — append-only
- `06-memory.md` (M) — mutable
- `state.json` — **single source of truth for execution state**
- `milestones/` directory with per-milestone subdirectories:
  - `M-{n}/plan.md`, `M-{n}/progress.md`, `M-{n}/findings.md`, `M-{n}/verification.md`, `M-{n}/reflection.md`
- `.agent/memory.md` (project-level shared memory)

## Phase 3: Milestone Execution Loop (m iterations)

For each Milestone M-i from 1 to m:

### M-i.1 O: Milestone Kickoff
1. Read frozen S and C
2. Read project-level `.agent/memory.md`
3. Select Features belonging to M-i
4. Write M-i execution plan to `milestones/M-i/plan.md`
5. Initialize Loop Guard counters in `milestones/M-i/plan.md`:
   - `planning_adjustments: 0 / 3`
   - `global_iteration: {current} / 50`
6. **Update state.json**:
   - `status: RUNNING`
   - `phase: MILESTONE_{i}`
   - `current_milestone: M-{i}`
   - `last_role: O`
   - `last_action: PLAN`
   - `last_result: OK`
7. Append to Audit Log: milestone start, features in scope, expected convergence k

### M-i.2 Feature Execution Loop (k iterations per Feature)

For each Feature F-j in M-i (in dependency order):

#### W: Worker Implementation
1. **Fresh Context Rule**: W starts with ONLY:
   - Feature spec from `04-features-and-milestones.md`
   - Frozen S and C (relevant assertions only)
   - Previous Handoff (H) if any
   - Shared state files (services.yaml, AGENTS.md if they exist)
2. W does NOT read previous worker trajectory, chat history, or implementation details of other features
3. Implement the feature
4. Run local checks (lint, type-check, unit tests for this feature only)
5. **Cm: Commit / Snapshot**
   - `git add` and `git commit` with message: `[task-{id}] F-{j}: {feature name}`
   - Record commit hash
   - If not a git repo, create a timestamped file snapshot in `.agent/tasks/{task-id}/snapshots/`
   - Cm is MANDATORY before Handoff; no exceptions
6. **Update state.json**:
   - `current_feature: F-{j}`
   - `last_role: W`
   - `last_action: IMPLEMENT`
   - `last_result: OK`
   - `budget.files_modified += count`

#### H: Structured Handoff
W MUST produce `milestones/M-i/handoff-F{j}.md` with exact sections:
```markdown
# Handoff: F-{j} — {Feature Name}

## Completed
- {exact deliverables}

## Remaining
- {what was scoped out or left for later}

## Commands Executed
| Command | Exit Code | Output Summary |
|---------|-----------|----------------|
| {cmd} | {code} | {summary} |

## Issues / Blockers
- {any problems encountered}

## Compliance Check
- [ ] Followed spec S requirement {X}
- [ ] Covered assertion ASSERT-{Y}
- [ ] Committed at {hash}
- [ ] Local checks passed

## Next Worker Notes
- {context the next worker needs}

---
Generated: {timestamp}
Worker: W-{id}
```
7. **Update state.json**:
   - `last_role: W`
   - `last_action: HANDOFF`
   - `last_result: OK`

#### V: Validator Inspection (Fresh Context)
1. **Fresh Context Rule**: V starts with ONLY:
   - Frozen Validation Contract C (assertions relevant to F-j)
   - Committed artifact at Cm hash (checkout or read snapshot)
   - Handoff H (for context, not for leniency)
   - V does NOT read worker trajectory, chat history, or implementation reasoning
2. Execute validation per assertion:
   - Scrutiny validation: lint, type-check, test suite, code review against contract
   - If applicable: user-testing validation (UI/E2E via screenshots or manual steps)
3. Record results in `milestones/M-i/verification-F{j}.md`:
   - Per-assertion status: PASS / PARTIAL / FAIL
   - Evidence collected
   - Issues found (blocking vs non-blocking vs suggestion)
4. **Self-Evaluation Bias Prevention**: V's verdict is final; W cannot dispute it
5. **Update state.json**:
   - `last_role: V`
   - `last_action: VALIDATE`
   - `last_result: PASS | PARTIAL | FAIL`

#### F: Fix Feature Generation (by O, not W)
1. If V reports FAIL or PARTIAL with blocking issues:
   - O reads verification report (NOT W)
   - O checks Fix Depth counter for F-j in `milestones/M-i/plan.md`
   - If Fix Depth < 3: O generates a Fix Feature F-fix-{n} with:
     - Root cause analysis
     - Target assertion IDs to satisfy
     - Specific fix instructions (not vague)
     - Increment Fix Depth counter
   - O appends F-fix to current Milestone plan
   - O records in Audit Log: fix generated, reason, linked assertion, fix depth
   - **If Fix Depth >= 3**: O does NOT generate another fix. Instead:
     - O marks F-j as PARTIAL with technical debt
     - O records debt in `milestones/M-i/reflection.md`
     - O proceeds to next Feature (or next Milestone if this was the last)
     - O appends to Audit Log: fix depth limit reached, auto-downgraded
   - **Update state.json**:
     - `fix_depth: {current}`
     - `last_role: O`
     - `last_action: GENERATE_FIX`
     - `last_result: OK | AUTO_DOWNGRADE`
2. W (or a new W context) executes F-fix following the same W→Cm→H→V loop
3. Factory benchmark: 34.4% fix feature ratio is normal and healthy

#### Convergence Check
- Repeat k until all features in M-i pass validation or k_max reached
- **k_max = 5** per Feature (including its Fix chain)
- If k reaches 5 and F-j still not fully PASS:
  - O auto-downgrades acceptance criteria to PARTIAL
  - O records technical debt
  - O proceeds to next Feature
  - O appends to Audit Log: k_max reached, auto-downgraded
  - **Update state.json**:
    - `current_iteration: 5`
    - `last_role: O`
    - `last_action: LOOP_GUARD`
    - `last_result: AUTO_DOWNGRADE`
- Factory data: median 2-4 validation rounds per milestone

### M-i.3 Milestone Gate
Before proceeding to M-(i+1):
1. O reviews all verifications in M-i
2. O increments `global_iteration` counter in state.json
3. If `global_iteration >= 50`:
   - O triggers **Graceful Termination** (see Phase 6)
   - O outputs all completed deliverables
   - O does NOT proceed to M-(i+1)
   - **Update state.json**: `status: COMPLETED`, `phase: GRACEFUL_TERMINATION`
4. **HUMAN CHECKPOINT 4** (if any FAIL or PARTIAL remains AND human intervention is explicitly requested by user):
   - Present: milestone status, remaining issues, proposed fix plan
   - Phrase: "Milestone {i} has N open issues. Proceed with fixes, or accept partial and continue?"
   - Note: O auto-downgrades by default; human checkpoint is optional override
5. If human says "accept partial", record technical debt in `06-memory.md`
6. O appends Milestone completion summary to Audit Log
7. **Update state.json**:
   - `completed_milestones: append M-{i}`
   - `completed_features: append all features in M-{i}`
   - `last_role: O`
   - `last_action: MILESTONE_GATE`
   - `last_result: OK`

## Phase 4: Audit Log (A) — Cross-Cutting Stream

A is NOT a phase; it is an append-only stream written by EVERY role after EVERY action.

Location: `.agent/tasks/{task-id}/05-audit-log.md`

Format per entry:
```markdown
| Timestamp | Role | Action | Target | Input | Output | Decision | Human |
|-----------|------|--------|--------|-------|--------|----------|-------|
| {ISO} | O | PLAN | M-1 | S+C | 3 features | waterfall chosen | - |
| {ISO} | W-1 | IMPLEMENT | F-1 | spec+handoff | code+tests | TDD approach | - |
| {ISO} | W-1 | COMMIT | F-1 | - | hash: abc123 | snapshot frozen | - |
| {ISO} | W-1 | HANDOFF | F-1 | - | handoff-F1.md | 2 issues noted | - |
| {ISO} | V-1 | VALIDATE | F-1 | C+Cm | 2 FAIL | fresh context eval | - |
| {ISO} | O | GENERATE_FIX | F-fix-1 | V report | fix spec | root cause: race cond | - |
| {ISO} | O | LOOP_GUARD | F-1 | fix depth=3 | auto-downgrade | depth limit reached | - |
| {ISO} | O | LOOP_GUARD | F-2 | k=5 | auto-downgrade | k_max reached | - |
| {ISO} | O | LOOP_GUARD | M-2 | planning=3 | freeze plan | plan limit | - |
| {ISO} | O | RESUME | task-123 | state.json | phase=M-2/F-3 | restored context | - |
| {ISO} | HUMAN | OVERRIDE | ASSERT-003 | "too strict" | relaxed | user request | YES |
```

Rules:
- Append-only; never edit or delete past entries
- Every git commit, every handoff, every validation must generate an audit entry
- **Loop Guard events must generate explicit audit entries** (auto-downgrade, strategy switch, graceful termination)
- **Resume events must generate explicit audit entries** with full state snapshot
- If human intervenes, record the exact command/prompt and O's response
- Project-level `.agent/audit-log.md` accumulates cross-task entries for trend analysis

## Phase 5: Memory & Final Delivery (M)

1. Cross-task memory merge:
   - Extract reusable patterns from `06-memory.md` into `.agent/templates/decomposition/{task-type}.md`
   - Extract anti-patterns into `.agent/templates/anti-patterns.md`
   - Extract assertion templates into `.agent/templates/contracts/{domain}.md`
2. Archive:
   - Keep: U, S, C, final plan, all handoffs, all verifications, audit log, memory, state.json
   - Optional cleanup: per-milestone working drafts (progress.md, findings.md) if storage is tight
3. Final Delivery Report (present to user) must include:
   - Task summary
   - Milestones completed: m_actual / m_planned
   - Convergence stats: avg k per milestone, total fix features, fix ratio, auto-downgrades triggered
   - Loop Guard events: fix depth limits, k_max reached, global iterations consumed
   - Assertion coverage: X/Y assertions fully satisfied
   - Audit summary: total actions, human interventions, token/effort estimate
   - Known limitations & technical debt
   - Reusable templates generated
4. **Update state.json**: `status: COMPLETED`, `phase: DELIVERED`, `last_role: O`, `last_action: DELIVER`

## Phase 6: Graceful Termination (Global Loop Guard)

When `global_iteration >= 50` or all milestones completed:

1. O stops accepting new features
2. O assembles all completed Cm snapshots into a coherent deliverable
3. O marks all uncompleted features as ABANDONED with reason (iteration limit / milestone skipped)
4. O generates final Audit Log summary
5. O merges Memory to project-level `.agent/memory.md`
6. O presents Final Delivery Report to user
7. **Update state.json**: `status: COMPLETED`, `phase: GRACEFUL_TERMINATION`
8. Task state: COMPLETED (even if partial)

## Phase 7: Suspend & Resume (User-Initiated)

### Suspend
When user says "pause", "stop for now", "continue later", or when 10-min stall detected:
1. O finishes current W→Cm→H→V→F cycle step (do not interrupt mid-action)
2. O writes `SUSPEND` entry to Audit Log with reason
3. **Update state.json**: `status: SUSPENDED`, `last_role: O`, `last_action: SUSPEND`, `last_result: OK`
4. O presents to user:
   ```
   Task suspended at: {phase}
   Current feature: {current_feature}
   Completed: {completed_milestones} / {total_milestones}
   To resume later, call: /universal-complex-task --resume {task-id}
   ```

### Resume
When skill is re-invoked and existing state detected (Phase 0.5):
1. O reads `state.json` and last 3 Audit Log entries
2. O verifies consistency: `state.json.phase` must match latest Audit Log action
3. If inconsistency detected → O warns user and asks for manual inspection
4. If consistent → O writes `RESUME` entry to Audit Log
5. **Update state.json**: `status: RUNNING`, `last_role: O`, `last_action: RESUME`, `last_result: OK`
6. O continues from exact state in `state.json` (same Milestone, same Feature, same Iteration)

## Loop Guardrails (Soft Constraints — O Self-Regulated)

These guardrails are **self-regulated by O** to prevent infinite loops WITHOUT breaking automation.
Humans are NOT interrupted unless the system is completely stalled (10-min rule).

| Trigger Condition | O Auto-Response | Human Interrupted? |
|-------------------|-----------------|-------------------|
| Fix depth >= 3 for same Feature | O marks PARTIAL, records debt, **proceeds** | No |
| Single Feature k >= 5 | O downgrades criteria, records debt, **proceeds to next Feature** | No |
| Planning adjustments >= 3 for same Milestone | O freezes plan, executes "imperfect but runnable" version | No |
| Global iteration n >= 50 | O triggers **Graceful Termination**, outputs best deliverable | No (reported at end) |
| Same root cause fails >= 2 consecutive times | O **must switch strategy** (Never Repeat Failures) | No |
| Zero progress for 10 minutes | O reports stall to user, **requests human intervention** | **Yes** |

### Design Rationale
- Factory's 34.4% fix ratio proves loops are healthy iteration, not errors
- Hard human interrupts would turn O into a "loop counter" and destroy automation
- Soft constraints let O act as the project manager's proxy: downgrade, skip, switch strategy
- Only **complete stagnation** (10 min no progress) warrants human attention

## Critical Constraints (Hard Rules)

| Rule | Violation Consequence |
|------|----------------------|
| **No Direct Communication** | W and V must not chat peer-to-peer; all state via files + O |
| **Serial Write** | Only one W writes at a time; no parallel file modifications |
| **Cm Before H** | Handoff without a commit hash is invalid; V must reject |
| **Fresh Context for W** | W must not read previous trajectory; only spec + handoff + shared state |
| **Fresh Context for V** | V must not read worker trajectory; only C + committed artifact |
| **V Never Self-Corrects** | If V finds issues, O generates F; W does not self-patch after V |
| **Contract Freeze** | Once C is confirmed, assertions cannot be added/removed to match implementation |
| **Dependency Freeze** | After Milestone 3 starts, no new inter-milestone dependencies |
| **Context Budget** | W reads max 5 files per feature; excess requires O approval |
| **2-Action Rule** | After 2 searches/reads, findings must be written to findings.md |
| **Audit Every Action** | Missing audit entry = action did not happen |
| **Loop Guard Self-Enforcement** | O must auto-downgrade at limits; waiting for human approval is a violation |
| **State.json Single Source of Truth** | O must update state.json after EVERY action; model must read it, never infer state |
| **Frozen File Protection** | `01-04` files are NEVER overwritten after confirmation, even during resume |

## Human Intervention Points (Mission Control Style)

Humans act as project managers, not code editors. Minimum intervention unit is plan-level reallocation.

| Trigger | Human Action | O Response |
|---------|------------|------------|
| S unclear | Revise scope | Refreeze S and C, replan features |
| Milestone stuck | "Re-assess and continue" | O re-evaluates remaining work, may split/skip features |
| Worker stuck | "Mark F-X complete and move on" | O accepts partial, records debt, proceeds |
| Direction change | "Drop X, add Y" | O replans remaining milestones, preserves completed Cm |
| Validation dispute | "Override V on ASSERT-Z" | Human can override with explicit audit entry; O adjusts contract |
| Budget exhausted | "Prepare MVP delivery" | O switches to conservative mode: only critical path features |
| **System stall (10 min)** | Any instruction | O reports stall cause, awaits human decision |
| **Suspend request** | "Pause here" | O finishes current step, writes SUSPEND state, reports resume command |
| **Resume request** | "Continue task-{id}" | O reads state.json, verifies consistency, resumes exact phase |

## Output Standards

- All files: Markdown
- Progress: `- [ ]` / `- [x]` task lists
- Task topology: Mermaid or indented lists in plan.md
- Audit: pipe-delimited table (machine-parseable)
- Handoff: exact 6-section format (mandatory)
- Loop Guard: every auto-downgrade must be recorded in both reflection.md and audit-log.md
- State: JSON, updated after every action, read before every action
