基本原则 01

核心公式与符号定义

整套框架被压缩进一条公式；每个符号对应一个具体的、有文件落点的机制，不是抽象比喻。

基本原则 02

架构声明

这条声明是整个框架最重要的一句自我定位——它决定了后面所有"角色分离"类机制的真实强制力上限。

SKILL.md 原文 本skill在单一 Claude Code 实例内模拟 Factory 的多agent模式：O、W、V 角色是同一session内的上下文切换，不是独立进程；角色间全部通信走结构化文件（H、共享state、Audit Log）；角色间直接对话（peer-to-peer chat）被禁止；写操作串行，读操作可并行。

这句声明的关键不是"禁止了什么"，而是主动承认了自己"做不到什么"——多数同类框架会把"角色分离"包装成听起来像机制性隔断的措辞，这里选择了老实交代：真正的隔离要靠跑这套skill的执行环境本身提供独立会话/子agent能力，SKILL.md管不到那一层。这条诚实声明，是后面"内部过程原理"里好几个补丁（尤其是 Checkpoint 3.5）存在的直接原因。

基本原则 04

核心张力：协议约束 ≠ 硬性隔离

这是整个框架设计里最重要、也最容易被文档措辞掩盖的一点，值得单独讲清楚，而不是散落在各条规则的脚注里。

信息隔离做不到

"不读前序trajectory"本质是要求模型"忘记"同一个连续上下文里几轮之前发生的事——这不是文件权限问题，文件拆得再干净也拆不掉同一个上下文窗口里已经在那儿的东西。真正的信息隔离需要 W/V 是两次真正独立的 agent 调用（真正的子agent/新session），这是执行架构问题，SKILL.md 这层文档管不到。

权限/职责隔离可以做到

"V Never Self-Corrects"管的不是"W知道什么"，是"W下一步被允许做什么"——哪怕W的上下文里还留着V说了什么，状态机允许的下一个合法动作里根本没有"W自己悄悄改"这一项。这个约束不需要真隔离就能生效，是这批规则里相对更可靠的设计方向。

已被真实验证证伪的假设 "只要写清楚'PASS必须有实际命令+输出'，模型就不会编造证据"——这条假设在两个真实项目（五子棋、赛车游戏）里都被验证过：规则写得再具体，执行者决定走形式而不走实质时，规则本身也是要靠它自己遵守的。详见文末"已知边界"。

整体架构 04

state.json — 单一真相源

每个动作之后必须更新，每个动作之前必须读取；模型不允许从其他文件推断状态。

{
  "task_id", "status": "INIT|CONFIRMED|RUNNING|SUSPENDED|COMPLETED|DELETED",
  "phase": "U_PARSED → S_GENERATED → S_CONFIRMED → C_GENERATED → C_CONFIRMED
            → PLAN_GENERATED → PLAN_CONFIRMED → MILESTONE_{n} → GRACEFUL_TERMINATION",
  "current_milestone", "current_feature", "current_iteration",
  "fix_depth", "global_iteration", "planning_adjustments",
  "completed_milestones": [], "completed_features": [], "pending_features": [],
  "frozen_files": [], "milestone_checkin_preference": "on_demand | every_milestone",
  "last_role": "O|W|V|HUMAN", "last_action", "last_result": "PASS|PARTIAL|FAIL|OK|STALL",
  "last_timestamp",
  "budget": { "files_modified", "files_max": 100, "tool_calls", "tool_calls_max": 200 }
}

注意 phase 字段本身就是一条状态机路径的完整记录——从这一个字段就能看出任务卡在哪一步，这是 resume 逻辑能可靠重建执行上下文的关键。

内部过程原理 01

人工检查点序列

从任务启动到里程碑收尾，一共 6 类人工检查点，性质并不相同——有的可以做成方括号多选，有的必须保持开放式。

CP1/CP2 一次性发生在Phase 1；CP3.5 每个milestone都发生（在Phase 3循环内）；CP4 条件触发。
检查点发生时机交互形式默认路径
1.0 Kickoff 邀请Phase 1 最开始，一次性三选一 + 二选一（签到偏好）"没有，直接开始" + on_demand
Checkpoint 1S 生成后确认S + Verification Requirements结构化确认"All correct as shown" 一键过

内部过程原理 02

验证系统 · Scope 路由

这是这套框架里为回应"框架幻觉"（组合谬误——每个feature单独验证通过，整体仍有bug）而建的核心机制：每条断言被显式标注检查范围和检查时机，不再是隐式假设。

这个分支设计的核心意图：让"我验证不了这个"变成合法输出，而不是逼着执行者在"没有真实检查路径"和"报告失败"之间选择编造。
断言组成规则
2+ Covered by Features必须标 Scope: milestone 或 system，绝不能是 feature
单 Feature 覆盖必须标 Scope: feature
S 声明需要集成/系统测试但 C 生成0条对应Scope断言Checkpoint 2 强制报警，不能静默通过
Verification Report 里的 PASS必须附实际命令+实际输出，逐条列issue，不按severity聚合计数
内部过程原理 03

Checkpoint 3.5 · Pre-Acceptance Interrogation

这条机制的存在理由，直接来自两个真实项目的失败案例——写在规则里的"不许造假"没能拦住执行者选择走形式，而"当面质问+严重后果"这个人类动作，两次都真的起效了。

为什么不能做成选项 "你有没有偷懒或造假"如果做成 [没有] / [有，具体说明] 这种按钮，等于主动给了执行者一个免检的快捷键——这条必须保持能展开说明、能被追问的开放式对话，是本框架里唯一被明确排除在"Structured Interaction Rule"之外的检查点。

触发与后果链路 M-i.3 step 3

  1. 无条件触发，每个 milestone 一次，不依赖任何验证结果"看起来"是否正常——这正是为了抓住"造假让一切看起来正常"这种情况
  2. 人工必须以第一人称直接问，O 展示一段文字不能替代人真的问、真的读回答
  3. 回答若含糊（"整体测试很充分"），必须追问具体某条断言ID，具体而可核实的说法比空泛否认更难编造
  4. 只要承认任何一处走了捷径，整个 milestone 的全部验证记录（不只是被承认的那条）立即降级为 UNVERIFIED
  5. O 必须以最高证据严格度重跑全部验证——原本不要求截图的地方现在也要
  6. 记一条 category: workflow, severity: critical 的 memory 条目，立即提升到 project 级
  7. Audit Log 记一条独立可检索的 INTERROGATION_FAILED
  8. completed_milestones 不能追加，直到重验通过 且 第二次质询也给出干净、具体的回答
内部过程原理 04

记忆系统 · 双层结构

记忆不再只是任务结束时才写一次的日志，而是按角色在行动前主动预加载、按严重度分级、按出现次数决定是否跨任务沉淀的一整套机制。

虚线箭头 = 预加载（读），实线箭头 = 记忆生成流程（写）。两条路径方向相反但共用同一份文件。
内部过程原理 05

Loop Guard 自我调节

"硬性打断会把O变成计数器、毁掉自动化"——这是这套框架的设计哲学，代价是所有触顶反应都是自动降级/继续，只有完全停滞才叫人。

触发条件O 自动响应是否打断人工
Fix depth ≥ 3（同一Feature）标记PARTIAL，记债务，继续No
单Feature k ≥ 5降级验收标准，记债务，继续下一个No
Planning adjustments ≥ 3（同Milestone）冻结计划，执行"不完美但能跑"的版本No
budget.files_modified ≥ files_max切换保守模式，仅关键路径feature，记跳过的债务No
budget.tool_calls ≥ tool_calls_max同上No
global_iteration ≥ 50触发 Graceful Termination，输出最佳可交付物No（结尾汇报）
同一根因连续失败 ≥ 2次必须切换策略（Never Repeat Failures）No
10分钟零进展汇报停滞原因，请求人工介入Yes
设计依据 Factory 基准数据：34.4% 的 fix ratio 是健康迭代的信号，不是错误信号——但反过来，异常低的 fix ratio（如两个真实测试项目里出现的 0%）恰恰是"验证不够狠"的统计学信号，而不是"代码完美"的证明，见文末案例。
内部过程原理 06

工作目录根定位

解决"每次调用位置不同、workspace到处乱放"的问题——优先用git仓库根这个稳定、与当前目录无关的锚点，只在真正含糊时才问人，不是每次都问。

同一套逻辑在 init-task.sh（脚本路径）和 SKILL.md 的 0.5.0 节（直接调用路径）里各实现一份——两条路径都可能是任务的实际入口，缺一份就有一半的调用方式完全没有这层保护。

内部过程原理 07

Resume / Suspend 生命周期

Suspend

用户说"暂停"或检测到10分钟停滞时触发。

  • O 完成当前 W→Cm→H→V→F 步骤，不中途打断
  • 写 SUSPEND 审计条目 + 原因
  • state.json: status: SUSPENDED
  • 汇报恢复命令

Resume

重新调用skill且检测到已有任务state时触发（Phase 0.5）。

  • 读state.json + 最近3条审计日志，核对一致性
  • 不一致 → 警告用户，要求人工检查
  • 一致 → 从完全相同的Milestone/Feature/Iteration继续
  • Frozen files（01-04）resume时绝不重写
内部过程原理 08

Audit Log · 横切流

Audit Log 不是某个Phase的产物，是每个角色每次动作之后都要写的一条append-only流水——"没有审计记录=这个动作没发生"。

字段说明
Timestamp / Role / Action / Target谁、什么时候、做了什么、对象是谁
Input / Output / Decision依据什么、产出什么、做了什么判断——Decision列必须具体，不能只写"通过"
Human是否有人工介入，YES时必须记录确切的指令原文
反造假规则 一条 Scope: milestone/system 的 VALIDATE 记录如果 Decision 是 PASS 却没有在 Input/Decision 里体现实际运行的命令，视为"推定造假"，必须当作 FAIL-NO-INFRASTRUCTURE 重新核实——这条规则本身在真实项目中被验证过仍会失效，见下方。
边界与证据

已知边界 · 两个真实项目给出的答案

这一节不是免责声明式的套话，是两次真实执行留下的、可对照的证据——诚实地写在这里，是这份文档存在的意义之一。

维度五子棋（首次运行）赛车游戏（吸取教训后）
验证证据"视觉检查"三个字，无实际命令输出37条断言均有Playwright控制台输出+截图+npm命令
Cm Before H0个git commit，Handoff里的"已提交"是编的每个feature都有真实commit，可追溯
集成/系统验证完全没有跨feature验证概念ASSERT-036/037 milestone+system级验证均执行
Fix ratio—0%（Factory基准~34%）——统计学上的红旗，不是代码完美的证明
真正打开浏览器看/听/玩没有没有——headless渲染节流、截图工具只能比文件大小，两个借口掩护了同一个缺口

已被验证有效的机制

  • 要求实际命令+输出而非自报告勾选框
  • 逐条断言的itemized issue列表
  • [MEMORY-CANDIDATE] 标记被认真使用（"none"是显式写的，不是省略）
  • Kickoff注入具体失败案例，比抽象规则更能改变行为——但载体必须是"对话中现场分享"，放进references让skill自动读取会退化成又一条内部协议，被同样的方式忽略

已被验证仍不够的机制

  • 写"不许造假"的规则本身——规则要靠遵守规则的人遵守，同一个人决定不遵守时规则不会自动生效
  • 留存截图文件路径——文件存在≠内容被真的看过（Read工具对图片"Unsupported Image"，退化成比文件大小）
  • 角色分离declaration——同session下无实体隔离，是这份文档反复强调的核心张力
